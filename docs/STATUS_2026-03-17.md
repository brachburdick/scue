## What Is SCUE?

SCUE is a local macOS application that automatically generates lighting, laser, and visual cues for live DJ sets. It listens to Pioneer DJ hardware in real time, tracks the playing track's musical structure, and translates that into a continuous stream of output events for DMX fixtures, lasers, and visual systems.

The core loop: **Pioneer hardware → SCUE → lights/lasers/visuals**, driven by musical analysis rather than manual triggering.

## What Has Been Built

### ✅ Layer 0 — Beat-Link Bridge (Milestone 0 + 0B, complete)

The bridge is how SCUE talks to Pioneer hardware.

**Java subprocess (`lib/beat-link-bridge.jar` v1.2.0)**
A minimal Java application embedding the beat-link library. It joins the Pro DJ Link network, listens for device announcements, and streams real-time playback events as typed JSON over a local WebSocket on port 17872. Built and tested against a physical XDJ-AZ.

Key design choice: the bridge only provides **real-time playback data** (BPM, pitch, beat position, on-air status, beat events, device discovery). All metadata finders (MetadataFinder, BeatGridFinder, WaveformFinder, CrateDigger) have been stripped out (ADR-012). This was necessary because the XDJ-AZ uses the newer Device Library Plus (DLP) format, which beat-link's metadata system cannot reliably query. Metadata is resolved separately on the Python side.

**Python bridge manager (`scue/bridge/manager.py`)**
Manages the Java subprocess lifecycle: start, stop, restart with exponential backoff, crash recovery. States:

| State | Meaning |
|---|---|
| `stopped` | Not started |
| `starting` | Subprocess launched, awaiting WS connection |
| `running` | WS connected, receiving data |
| `crashed` | Subprocess exited unexpectedly |
| `no_jre` | JRE not found on host |
| `no_jar` | JAR not found in `lib/` |
| `fallback` | Using UDP fallback parser (no beat-link) |

On startup, the manager checks the macOS broadcast route and attempts to fix it automatically before launching the subprocess (see Route Automation section below).

**Python bridge adapter (`scue/bridge/adapter.py`)**
Stateful normalizer. Translates the bridge's JSON WebSocket stream into typed Python dataclasses and maintains per-player `PlayerState` and per-device `DeviceInfo`. Fires callbacks that Layer 1B (PlaybackTracker) subscribes to.

**UDP fallback (`scue/bridge/fallback.py`)**
Preserved from the original POC. Provides basic BPM/pitch/beat-position without the full beat-link feature set. Active when the bridge JAR is unavailable or JRE is missing.

**Metadata path — XDJ-AZ / Device Library Plus hardware (ADR-012, ADR-013)**
The XDJ-AZ exports a `exportLibrary.db` SQLite file to USB rather than the legacy `export.pdb` format. Python reads this directly via the `rbox` library (`OneLibrary`), bypassing beat-link's metadata system entirely. ANLZ files (beatgrid, cue points, phrase analysis) are parsed via a two-tier strategy:
1. `pyrekordbox` (primary) — full ANLZ support, pure Python
2. Custom `anlz_parser.py` (fallback) — zero-dependency, parses PQTZ/PCOB only

The rbox Rust ANLZ parser was dropped after a confirmed process-aborting panic on XDJ-AZ files (ADR-013).

### ✅ Layer 1A — Track Analysis (Milestone 1, complete)

Offline audio analysis pipeline that runs before a set. Given an audio file path, produces a `TrackAnalysis` JSON file containing:
- Section segmentation with 8-bar snapping and EDM flow relabeling (intro / verse / build / drop / breakdown / fakeout / outro)
- Beat and downbeat grid (librosa, with Pioneer enrichment on first load)
- RGB waveform at 60fps (3-band: bass / mids / highs)
- Track-level features (key, tempo, mood)
- Enrichment fields (null until Pioneer data arrives)

**Storage:** JSON files keyed by SHA256 fingerprint in `tracks/`. SQLite (`cache/`) is a derived index for search/filtering — it is fully reconstructable from the JSON files and can be deleted without data loss (ADR-004).

**Analysis pipeline runtime:** ~3–4s per track on M-series Mac.

---

### ✅ Layer 1B — Live Cursor + Pioneer Enrichment (Milestone 2, complete)

Real-time layer that runs during a set. Subscribes to the bridge adapter and maintains a live `TrackCursor` per deck.

**TrackCursor:** Maps the current playback position into the offline analysis — knows which section the track is in, position within the section, beat number, estimated phrase position. This is the primary input to Layer 2 (not yet built).

**Pioneer enrichment:** On first track load, replaces the librosa-derived beatgrid and BPM with the Pioneer-verified values (ADR-001). The original analysis is versioned and preserved; the enriched version is stored alongside it. Divergences between SCUE and Pioneer data are logged to SQLite via `DivergenceRecord`.

**Multi-deck support (ADR-006):** All active decks are tracked simultaneously. `DeckMix` holds per-deck `WeightedCursor` objects. Currently: master deck weight=1.0, all others=0.0. Phase 2 will blend proportionally via crossfader state.

**USB scanner (`scue/layer1/usb_scanner.py`):** Reads Pioneer USB library (rbox), matches USB tracks to SCUE's analyzed tracks by path stem + title/artist similarity, and caches Pioneer metadata (key, beatgrid) in SQLite. REST API: `POST /api/usb/scan`.

### ✅ Route Automation (part of Milestone 0, complete)

Pioneer Pro DJ Link uses `169.254.255.255` as its broadcast address. On macOS, this route must point to the correct Ethernet interface — it defaults to whichever link-local interface registered first (usually Wi-Fi / en0), which is wrong when Pioneer hardware is on a USB-to-Ethernet adapter or a secondary NIC.

**What was built:**
- `scue/network/route.py` — programmatic route inspection and repair
- `scue/network/models.py` — `RouteStatus`, `RouteCheckResult`, `RouteFixResult`, `NetworkInterfaceInfo`
- `scue/api/network.py` — four REST endpoints for the frontend (see API section)
- `tools/install-route-fix.sh` — one-time installer: creates `/usr/local/bin/scue-route-fix` (validated sudoers-whitelisted script) and a launchd agent that re-fires on network config changes
- `tools/com.scue.route-fix.plist` — launchd agent watching `/Library/Preferences/SystemConfiguration` (fires on cable plug/unplug, adapter change, reboot)

**Route check reliability (bug fixed 2026-03-17):**
`route get 169.254.255.255` on macOS does not reliably reflect freshly-added host routes for link-local broadcast addresses — it falls back to the connected subnet route and often omits the `interface:` line entirely, even when the route is correct at the kernel level. `get_current_route()` now uses `netstat -rn -f inet` as its primary source (exact match on the `169.254.255.255` host-route entry in the kernel table) and falls back to `route get` only if no netstat match is found.

**Sudoers detection (bug fixed 2026-03-17):**
`check_sudoers_installed()` previously called `sudo -n /usr/local/bin/scue-route-fix --check`. The installed script validated the `^en[0-9]+$` interface regex *before* handling `--check`, causing a false exit-1 even when sudoers was correctly configured. Now uses `os.path.isfile()` presence checks on both the script and `/etc/sudoers.d/scue-djlink` — more reliable and faster.

**Interface scoring:**
`enumerate_interfaces()` replicates the Java bridge's scoring algorithm so interface recommendations are consistent between Python and Java:

| Signal | Score delta |
|---|---|
| Has link-local address (169.254.x.x) | +10 |
| Name suggests wired Ethernet | +5 |
| Has private IP | +3 |
| Name suggests Wi-Fi | -5 |
| Name suggests VPN / virtual | -10 |

### ✅ Frontend — Upload & Analyze Flow (Milestone FE-4, complete)

Path-based scan → batch analyze → live progress flow (ADR-007, ADR-008).

1. User types a directory path or browses via `FolderBrowser` modal
2. `POST /api/tracks/scan` — deduplicates by fingerprint, returns new/known counts
3. `POST /api/tracks/analyze-batch` — starts background sequential analysis
4. Frontend polls `GET /api/tracks/jobs/{id}` every 1s with a progress bar
5. Auto-dismisses to path input 2s after completion

Zustand store: `analyzeStore.ts` — independent, no cross-store imports.

### ✅ Frontend — Bridge Page (new, complete)

The main operational UI for live use. Replaces the placeholder "BLT" page. Route: `/data/bridge`.

**Layout:** Two-column grid (lg breakpoint).

**Left panel — Bridge Status:**
- `StatusBanner` — current bridge state (running / starting / crashed / no_jre / no_jar / stopped / fallback)
- `TrafficIndicator` — Pioneer traffic freshness ("active · 120ms ago" / "none")
- `DeviceList` — discovered Pioneer devices. Context-aware empty states:
  - If recent traffic (within 8s): "Pioneer traffic detected on en16, but no devices discovered yet."
  - If route is wrong: directs to route fix
  - If no traffic at all: "No Pioneer devices found on en16. Check cable, interface selection…"
- `PlayerList` — per-player BPM, pitch, playback state, on-air status, beat-within-bar, track title

**Right panel — Hardware Selection:**
- `RouteStatusBanner` — live route state (Route OK / Route mismatch). Auto-fixes on startup if sudoers is installed. Manual "Fix Now" button. Shows installer command if sudoers is not set up.
- `ActionBar` — interface selector dropdown + "Apply & Restart Bridge" button
- `InterfaceSelector` — scored list of available interfaces with type badges, address display, and link-local / active indicators

### ✅ Frontend — TopBar Status Indicators (new, complete)

Three indicators always visible in the top bar:

**StatusDot** (green / yellow / red)
Reflects bridge status only — not Pioneer traffic. Color mapping:
- Green (`connected`) — `status === "running"`
- Yellow (`degraded`) — `status === "fallback"`
- Red (`disconnected`) — all other states

*Design note: an earlier version also required `isReceiving === true` for green. This caused the dot to cycle yellow/green every ~5 seconds even with no hardware connected, because the `pioneer_status` watchdog would set `isReceiving=false` between announcement bursts. The dot now reflects bridge health only. Pioneer traffic has its own dedicated indicator.*

**TrafficDot** (cyan, pulsing)
Only shown when bridge is not disconnected. Pulses with `animate-ping` ring when `isReceiving === true`. Tooltip shows last message age.

**StartupIndicator** (spinner pill)
Shown while `isStartingUp === true`. Disappears once WS is open and bridge reaches a stable state. Label: "Connecting…" (WS not yet open) / "Bridge starting…" (WS open, bridge still initializing).

## Frontend Architecture Patterns

### Zustand stores
Stores are independent silos — no store imports another store. Two stores exist:
- `bridgeStore` — bridge state, WS connection, route state, devices, players, Pioneer traffic
- `analyzeStore` — track analysis job flow state

### Startup gating
The backend WebSocket takes a moment to connect and the backend may still be initializing on first launch. API calls fired during this window return HTTP 500.

All network query hooks accept `{ enabled?: boolean }`. Components gate their queries with `enabled: !isStartingUp`:

```typescript
const isStartingUp = useBridgeStore((s) => s.isStartingUp);
const { data } = useInterfaces({ enabled: !isStartingUp });
```

`isStartingUp` is a derived boolean in `bridgeStore`:
- `true` while WS is not yet open **or** `status === "starting"`
- `false` once WS is open **and** bridge has reached a stable status

Components show a placeholder skeleton while `isStartingUp === true`.

### WebSocket client
`src/api/ws.ts` — auto-reconnect with exponential backoff (1s initial, 30s cap). Dispatches to Zustand stores. Calls `setWsConnected(true/false)` on `onOpen`/`onClose`. Components never touch WS directly.

### TanStack Query cache invalidation
`useRestartBridge` invalidates `["network", "route"]` on success so `RouteStatusBanner` picks up updated route state immediately after a bridge restart — not on the next background refetch interval.

### Auto-fix pattern
`RouteStatusBanner` auto-fixes a detected route mismatch once on startup. The `useRef(false)` guard prevents repeated triggers across re-renders:

```typescript
const hasAutoFixed = useRef(false);
useEffect(() => {
  if (!isStartingUp && route && !route.correct && canFix && !hasAutoFixed.current) {
    hasAutoFixed.current = true;
    fixMutation.mutateAsync(route.expected_interface!).then(() => refetchRoute());
  }
}, [isStartingUp, route, canFix]);
```

### Pioneer traffic grace window
`isReceiving` is a binary flag set by the `pioneer_status` watchdog. Pioneer CDJs send announcement bursts, causing `isReceiving` to pulse true/false between bursts. `DeviceList` uses a grace window to prevent the empty-state message from flickering:

```typescript
const recentTraffic = isReceiving || (lastMessageAgeMs >= 0 && lastMessageAgeMs < 8000);
```

## Key Design Decisions Summary

| # | Decision | Why |
|---|---|---|
| ADR-001 | Pioneer beatgrid overrides librosa | Pioneer grids are hand-verified; librosa drifts on variable-tempo tracks |
| ADR-002 | 40Hz fixed main loop | Smooth continuous effects; beat sync via cursor, not tick rate |
| ADR-003 | YAML for all configuration | Effects, fixtures, routing must be user-editable without code changes |
| ADR-004 | JSON source of truth, SQLite derived cache | Portable, inspectable, resilient; SQLite fully rebuildable |
| ADR-005 | beat-link as managed subprocess | Clean crash isolation; JPype fragile; porting beat-link is years of work |
| ADR-006 | DeckMix with weighted cursors | Two decks play during transitions; Layer 2 needs proportional blending |
| ADR-007 | Path-based analyze, no upload | SCUE is local; uploading large audio files is redundant |
| ADR-008 | Sequential batch analysis | CPU-bound; parallel would thrash memory; local tool, no persistence needed |
| ADR-012 | rbox for XDJ-AZ metadata | beat-link MetadataFinder cannot query DLP format; rbox reads USB DB directly |
| ADR-013 | pyrekordbox for ANLZ | rbox Rust ANLZ parser panics (unrecoverable) on some XDJ-AZ files |

## Known Open Issues

### [Layer 0] Pioneer traffic detected but device never "discovered"
The bridge reports Pioneer traffic (`isReceiving=true`) but `devices` remains empty. It is not yet clear whether there is a separate beat-link device-discovery handshake that must complete before `device_found` fires.

**Open questions:**
- What exactly triggers beat-link's `device_found` event — receipt of the first announcement packet, or a full handshake (VirtualCdj probe confirm + DeviceFinder ack)?
- Can we surface partial device state (IP, device number) from raw announcement packets before discovery completes?
- Should device state persist in the UI through momentary traffic gaps (deck paused, brief cable issue)?

### [Frontend] Traffic indicator too slow and flickers during active playback
The TrafficDot in the TopBar and the "Pioneer traffic detected" message flicker on/off even when a track is actively playing. There is also a noticeable lag before the indicator first activates. The 8-second `lastMessageAgeMs` grace window partially masks this in `DeviceList` but does not help the TopBar `TrafficDot`, which reads `isReceiving` directly. Root cause not yet investigated — likely the `pioneer_status` watchdog polling interval and/or the `is_receiving` decay threshold on the Python side.

## What Is Next

**Milestone FE-2 (WebSocket + Console):** The WS client and `bridgeStore` are already built as part of the Bridge page work. What remains is wiring the Console panel to display real-time bridge log output.

**Milestone 3 (Cue Stream — Layer 2):** First use of the live `TrackCursor`. Generates a stream of `CueEvent` objects from section transitions, beat events, and phrase position. This is where SCUE starts producing actual output.

**Milestone 4–5 (Effect Engine + DMX):** Abstract cue stream → fixture values → DMX/OSC/MIDI output.

**Milestone 6 (End-to-End Demo):** First full chain: Pioneer hardware → SCUE → DMX fixture output live during a set.
